<remaining_work>

**Bug 1 — Arrow labels too small (implement first)**
Current `makeArrowLabel` uses QLabel with `setFixedHeight(22)` and `setPointSize(18)` bold.
18pt ≈ 24px, clipped by 22px height. Also no fixed width — label may expand beyond 22px.
Fix: Replace with `ArrowLabel` custom widget class using `setFixedSize(22, 22)` and
`paintEvent` (QPainter). Same approach as ChannelButton. Direction parameter: Down/Right.
File: `src/ShuffleMatrixWidget.h` and `src/ShuffleMatrixWidget.cpp`.

**Bug 4 — Green/blue column gap (likely fixed by Bug 1)**
User sees: `red  green  [small gap]  blue  alpha  [large gap]  0  1  [small gap]  r  g  [gap]  b  a`
The [small gap] within in1 is most likely caused by unconstrained-width arrow QLabels pushing
one column wider. Fixing Bug 1 should fix this.
Additional fix needed: move `_in2Picker` widget placement to start at `in2StartCol` (column
`in1Count+4`) instead of `sepBeforeConstCol` (column `in1Count`), so the two 8px separator
columns remain visually present rather than being absorbed by the picker's span.

**Bug 5 — Disabled buttons invisible (simple, do second)**
`paintEvent` uses `QColor(55, 55, 55)` for disabled buttons. On Nuke's dark panel background
(~#2b2b2b) this is invisible. Fix: change to `QColor(85, 85, 85)` fill and `QColor(110, 110, 110)`
border in `ChannelButton::paintEvent` (in `src/ShuffleMatrixWidget.cpp`).

**Bug 3 — ChannelSet picker requires two interactions (diagnostic first)**
Hypothesis: `set_text()` on a ChannelSet knob does not immediately update the Op's C++
backing member (`_in1ChannelSet`). So `setChannelSets(_in1ChannelSet, ...)` in `knob_changed`
passes the OLD value on the first interaction.
Diagnostic: Write `DeepCTest` node in `src/DeepCTest.cpp`:
  - `ChannelSet_knob` named "layer"
  - `String_knob` named "result" (write resolved channel names here)
  - `Bool_knob` named "flag" (flip each time "layer" changes)
In `knob_changed`: if `k->is("layer")`, flip flag and write channel names to result.
Add to `CMakeLists.txt` as separate target. Test in Nuke to confirm whether backing variable
is synchronously updated. Then fix the actual picker update path accordingly. Remove after.

**Bug 2 — Checks cleared on ChannelSet change (largest, do last)**
USER DECISION: Change state format from channel-name-based to positional indexing.
Current format: `"rgba.red:in1:rgba.red,rgba.green:in1:rgba.green,..."`
New format: `"out1:0:in1:2,out1:1:in1:1,out2:0:const:0,..."`
  (outputGroup : outRowIdx : sourceGroup : sourceColIdx)
Changes required:
1. `src/ShuffleMatrixWidget.cpp` buildLayout() — button objectNames change to
   `"out1|0|in1|2"` format (group|rowIdx|sourceGroup|colIdx)
2. `src/ShuffleMatrixWidget.cpp` syncFromKnob() — state restore by position
3. `src/ShuffleMatrixKnob.cpp` from_script() — add migration: detect old format
   (contains "rgba." or "in1:" etc), silently clear state
4. `src/DeepCShuffle2.cpp` _validate() — resolve positions to channel names using
   current ChannelSets; helper: channelAtIndex(ChannelSet, int) → Channel

<context>
We've been through 3 rounds of UAT fixes where agents were guessing without being able to
run Nuke. The pattern of "fixed but still wrong" led to this session where we read the code
together and identified root causes properly.

The state redesign (Bug 2) is the fundamental architectural fix — the decision to store
positions instead of channel names is correct and will eliminate the whole class of "routing
disappears when layer changes" bugs. It's also the riskiest change because it touches the
serialization format (breaks old scripts) and the engine parsing.

The arrow/gap issue (Bug 1/4) is most likely ONE root cause: unconstrained QLabel widths
in makeArrowLabel pushing QGridLayout columns apart unpredictably. The custom ArrowLabel
with setFixedSize(22,22) is the clean fix.

The disabled-invisible issue (Bug 5) is purely a color contrast problem. Trivial fix.

The double-update lag (Bug 3) is the most uncertain — could be set_text() timing, could be
something else entirely. The DeepCTest diagnostic will settle it.
</context>
