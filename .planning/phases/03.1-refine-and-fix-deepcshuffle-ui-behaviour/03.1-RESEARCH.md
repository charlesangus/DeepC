## User Constraints (from CONTEXT.md)

### Locked Decisions

**Const 0 / 1 Source Columns**
- Add const:0 and const:1 as toggle columns in the matrix widget
- Column position: in1 channels | 0 | 1 | in2 channels (matching reference)
- Column headers labeled "0" and "1"
- Engine already supports const:0 / const:1 in `_validate()` — only the widget needs adding

**Default Identity Routing**
- When `_matrixState` is empty (new node, first open), initialize identity routing: each output channel maps to its matching input channel by name (e.g. `rgba.red:in1:rgba.red`)
- Identity initialization happens in `buildLayout()` when state is empty and valid ChannelSets are present

**Routing State Persistence on ChannelSet Change**
- `_matrixState` is never cleared or modified when ChannelSets change
- The widget rebuilds columns/rows to reflect the new ChannelSets, then restores checked state by name-matching entries in `_matrixState`
- If a channel is no longer present (ChannelSet changed), its button is simply not rendered — state entry preserved silently
- If the ChannelSet is restored, the previously-stored routing is restored automatically (name match)
- No reset, no pruning — the state is sticky

**Radio Enforcement Bug — Two Fixes Required**
1. Clicking a checked cell = no-op: In `onCellToggled`, if `checked == false`, do nothing (return early). Rows must never be left with no source selected.
2. out1 and out2 are independent: The row enforcement key must incorporate the output group. Use `outGroup:channelName` (e.g. `"out1:rgba.red"` vs `"out2:rgba.red"`) as the radio constraint scope — not bare `channelName`. Buttons in out2 rows must not affect out1 rows and vice versa.

**"None" ChannelSet — Disabled, Not Hidden**
- When a ChannelSet is set to `none` (Chan_Black), its matrix buttons remain visible but are disabled (greyed out, not clickable)
- Routing state for that ChannelSet's rows/columns is preserved in `_matrixState`
- This applies to both in-columns (in2=none → in2 columns disabled but shown) and out-row groups (out2=none → out2 rows disabled but shown)

**ChannelSet Change — Instant Update (No Second-Change Required)**
- Matrix must rebuild immediately on the first ChannelSet change
- Root cause: likely `KNOB_CHANGED_ALWAYS` or `showPanel` trigger not firing on first change; investigate whether `knob_changed` → `setChannelSets` → `updateWidgets` chain fires correctly on initial panel open and on each ChannelSet change event
- Fix must ensure single-step responsiveness

**Knob Positioning — Embed Pickers in Widget**
- Remove the four standard NDK `Input_ChannelSet_knob` calls from `knobs()` (or hide them from the panel)
- Embed in1 and in2 ChannelSet pickers as widget-internal controls above their respective column groups (matching reference top-area layout)
- Embed out1 and out2 ChannelSet pickers as widget-internal controls to the right of their respective row groups (matching reference right-side layout)
- The embedded pickers must: (a) read current ChannelSet state from the Op, and (b) write back to the Op's ChannelSet knobs when changed, triggering `knob_changed`

**Visual Style — Colored X Checkboxes + Arrows**
- Channel color map (Nuke standard):
  - R → red
  - G → green
  - B → blue
  - A → grey/white
  - Z → red (same as R)
  - U, V → different magenta variants
  - X, Y, Z, W → different magenta/purple variants
  - Other channels → reasonable neutral color
- Checked state: button shows a styled X mark (not just highlight/background change)
- Unchecked state: button shows the channel color without X
- Disabled state (ChannelSet = none): buttons visually greyed out, distinct from active-unchecked
- Arrows:
  - Down-arrows below column header labels (pointing down toward data rows)
  - Right-arrows on each output row label (pointing right)
  - Matching the reference visual

- Exact Qt stylesheet / custom paint approach for X-mark checkboxes
- Whether to use `QPushButton::paintEvent` override or a stylesheet with pseudo-states
- Exact color hex values for U/V/X/Y/Z/W channels (beyond the clearly-stated ones above)
- How to handle channels with no obvious color assignment

## Summary

This phase is a targeted bug-fix and visual-polish pass on the DeepCShuffle2 / ShuffleMatrixWidget implementation produced by Phase 3. The engine (`_validate`, `doDeepEngine`, `getDeepRequests`) requires no changes — all work is in the Qt widget and the Op's knob/panel wiring.

The current implementation (screenshot: `reference/Screenshot_20260315_140709.png`) has the correct structural layout but is missing: const:0/const:1 columns, colored X-mark buttons, down/right-arrow decorations, embedded ChannelSet pickers (currently shown as separate NDK knobs above the matrix), proper radio enforcement scoping across out1/out2 groups, the no-op-on-uncheck guard, and the disabled-not-hidden behavior for none ChannelSets.

The reference image (`reference/shuffle.png`) shows all of these features in the built-in Nuke Shuffle2 node. The goal of this phase is exact visual and behavioral parity with that reference.

**Primary recommendation:** Work through the seven locked decisions in order — radio bug fixes first (lowest risk, isolated to `onCellToggled`), then const columns, then disabled-not-hidden, then identity routing init, then visual styling (colored X + arrows), then embedded pickers (highest coupling change), then knob-change responsiveness fix.

## Architecture Patterns

### Affected Files
```

├── ShuffleMatrixKnob.h/.cpp # no changes expected
├── ShuffleMatrixWidget.h    # possibly add outGroup tracking members
└── ShuffleMatrixWidget.cpp  # most changes: buildLayout, onCellToggled, syncFromKnob
```

### Pattern 1: Radio Enforcement with outGroup Scope
**What:** Each row in the matrix is a radio group. The scope of the radio group must be `outGroup:channelName`, not bare `channelName`, because out1 and out2 can have identically-named channels that must be independent.

**Current bug:** `onCellToggled` compares `otherOutput == outputChannelName` — this is a bare channel name like `"rgba.red"`. When out2 also has `"rgba.red"`, its button matches and gets incorrectly unchecked.

**Fix:** During radio enforcement loop in `onCellToggled`, additionally compare `outGroup` (the group the current row belongs to). Since `outputChannelName` alone is ambiguous, the calling lambda must also capture which group the row belongs to. The simplest approach is to capture the group name alongside the output name in the `connect` lambda, and pass it as a fourth parameter (already half-done — `inputGroup` is the column group, not the row group; the row group needs separate capture).

**Concrete approach:**
- In `buildLayout`, when constructing the `connect` lambda for each button in the out1 rows, capture `"out1"` as `capturedOutGroup`. For out2 rows, capture `"out2"`.
- The `onCellToggled` signature already has `inputGroup` for the column group; add a `outputGroup` parameter (or incorporate it into the existing signal signature).
- In radio enforcement, the scope key is `capturedOutGroup + ":" + capturedOutput`.

```cpp
// In buildLayout, out1 data rows:
const std::string capturedOutGroup = groupLabel; // "out1" or layer name
connect(btn, &QPushButton::toggled,
    [this, capturedOutGroup, capturedOutput, capturedSource](bool checked) {
        this->onCellToggled(capturedOutGroup, capturedOutput, "in1", capturedSource, checked);
    });
```

### Pattern 2: No-Op on Uncheck (Mandatory Check Guard)
**What:** In `onCellToggled`, if `checked == false` (user clicked an already-checked button, Qt unchecks it), return immediately. Rows must always have exactly one source selected.

```cpp
void ShuffleMatrixWidget::onCellToggled(...)
{
    if (!_knob) return;
    if (!checked) return;  // no-op: rows must never be left unrouted
    // ... rest of existing logic
}
```

### Pattern 3: Const Columns (0 and 1)
**What:** Two additional source columns between in1 and in2. The column headers are "0" and "1". The button objectName uses source name `"const:0"` and `"const:1"`.

**Column layout after change:**
```
Cols 0 .. in1Count-1           : in1 toggle columns
Col  in1Count                  : "0" const column
Col  in1Count+1                : "1" const column
Col  in1Count+2                : visual spacer between const and in2
Cols in1Count+3 .. +in2Count   : in2 toggle columns (always shown, disabled if in2=none)
Col  outLabelCol               : output channel name / group name label
```

The const columns are always present regardless of ChannelSet state. They are never disabled.

### Pattern 4: Disabled-Not-Hidden for None ChannelSets
**What:** When `in2ChannelSet == Chan_Black`, the in2 columns must still be rendered but with buttons in disabled state. Same for out2 rows.

**Current code:** `if (in2Active)` / `if (out2Active)` guards skip the columns/rows entirely. Remove these conditional skips; always render columns and rows, but call `btn->setEnabled(false)` when the corresponding ChannelSet is Chan_Black.

**Disabled state distinction:** Qt's disabled state automatically greys the button appearance. Combine with QSS disabled pseudo-state for explicit color if needed.

### Pattern 5: Identity Routing Initialization
**What:** In `buildLayout()`, after constructing the layout, if `_knob->matrixState()` is empty AND the ChannelSets have channels, write an initial identity state.

```cpp
// At end of buildLayout(), after layout is fully constructed:
if (_knob && _knob->matrixState().empty())
{
    // Build identity: each out1 channel -> matching in1 channel by name
    std::ostringstream identityState;
    bool firstEntry = true;
    for (const std::string& outputChannelName : out1Rows)
    {
        for (const std::string& sourceChannelName : in1Columns)
        {
            if (outputChannelName == sourceChannelName)
            {
                if (!firstEntry) identityState << ',';
                identityState << outputChannelName << ":in1:" << sourceChannelName;
                firstEntry = false;
                break;
            }
        }
    }
    // Do NOT call _knob->setValue() from buildLayout — that would fire changed()
    // recursively. Instead, write directly to the state and re-sync buttons.
    // Use a separate helper or call syncFromKnob() after setting via internal path.
}
```

**Important:** Identity init must not trigger a recursive rebuild. Use a direct write to `_matrixState` via `_knob` without firing `changed()`, then call the sync logic inline. Alternatively, write the identity state before calling `buildLayout()` from `syncFromKnob()` — check if state is empty at that point and populate it first.

### Pattern 6: Visual Styling — Colored X Buttons + Arrows
**What:** QPushButton styling to reflect channel color and checked/unchecked/disabled states.

**Recommended approach — QSS on individual buttons:**
```cpp
// Color map
static QColor channelColor(const std::string& shortName)
{
    if (shortName == "red"   || shortName == "r" || shortName == "z") return QColor(180, 40, 40);
    if (shortName == "green" || shortName == "g")                     return QColor(40, 140, 40);
    if (shortName == "blue"  || shortName == "b")                     return QColor(40, 80, 180);
    if (shortName == "alpha" || shortName == "a")                     return QColor(160, 160, 160);
    if (shortName == "u")                                             return QColor(160, 40, 160);
    if (shortName == "v")                                             return QColor(200, 40, 200);
    // fallback
    return QColor(90, 90, 90);
}
```

For X-mark on checked state, the cleanest approach that avoids QSS text rendering issues is a `QPushButton` subclass with a custom `paintEvent` that draws a filled color rect + an "X" drawn via two `QPainter::drawLine` calls when `isChecked()`. This avoids QSS `:checked { content: "X" }` which does not exist in Qt — text on pseudo-states requires `text()` to be set, which overrides the button's normal empty appearance.

Alternatively, set `btn->setText("X")` at checked time and `btn->setText("")` at unchecked time — but this requires syncing text with state changes, adding bookkeeping.

**Simplest implementation:** custom `QPushButton` subclass `ChannelButton` with:
- Constructor takes `QColor baseColor`
- `paintEvent`: fills background with `baseColor` (dimmed if disabled), draws centered "X" in white/black using `QPainter::drawLine` when `isChecked()`
- No QSS needed at all

**Arrows:** Use `QLabel` with Unicode arrow characters:
- Down-arrow below column header: `QLabel("↓")`
- Right-arrow on row output label: prepend `"→ "` to the output channel label text, or use a separate `QLabel("→")` to the left of the output name label

### Pattern 7: Embedded ChannelSet Pickers
**What:** Replace the four `Input_ChannelSet_knob` rows in the panel with pickers embedded inside the matrix widget itself.

**Two implementation paths:**

**Path A — Hide NDK knobs, use QComboBox:**
- In `knobs()`, keep the four `Input_ChannelSet_knob` calls but add `SetFlags(f, Knob::INVISIBLE | Knob::DO_NOT_WRITE | Knob::NO_UNDO)` — this keeps the knobs as the source of truth for serialization while hiding them from the panel.
- In `buildLayout()`, create `QComboBox` widgets populated with channel layer names from the NDK channel table.
- When the user selects a layer in the QComboBox, call `knob("in1")->set_value(...)` via `_knob->op()->knob(...)` to write back to the hidden knob, then trigger `knob_changed`.

**Path B — Delegate label that opens NDK picker:**
- In `buildLayout()`, create `QPushButton` or `QLabel` showing the current ChannelSet name. When clicked, find the corresponding hidden NDK knob widget and call `showPanel()` on it or simulate a mouse event to open its picker popup.
- This is fragile and not recommended.

**Recommended: Path A.** NDK's `Knob::INVISIBLE` (or equivalent flag) hides the knob row from the panel while keeping the knob alive for serialization and `knob_changed` routing. The QComboBox is populated by iterating over `DD::Image::channel_names()` or walking the `ChannelSet` of the connected input.

**Reading back current state:** The knob's current value is already pushed into `ShuffleMatrixKnob` via `setChannelSets` in `knob_changed`. The QComboBox initial selection can be set from `_knob->in1ChannelSet()`, reading the layer name.

**Writing back changes:** From the QComboBox `currentTextChanged` signal, call back to the Op to update the ChannelSet knob. One reliable way: `_knob->op()` returns the `Op*`; cast to `DeepCShuffle2*`; call a new public method `setIn1Layer(const char*)`. Alternatively, directly call `_knob->op()->knob("in1")->set_text(layerName)` followed by `_knob->op()->knob("in1")->changed()`. The exact NDK API for programmatically changing a ChannelSet knob value must be verified — see Pitfalls section.

### Pattern 8: ChannelSet Change Single-Step Responsiveness
**What:** The current `knob_changed` fires on `k->is("in1")` etc. and calls `setChannelSets` + `updateWidgets`. The bug report is that the first change requires a second interaction.

**Likely root cause:** On the first `showPanel` call, `knob_changed` fires with `k == &Knob::showPanel` — the `setChannelSets` + `updateWidgets` chain runs. But the widget has not been constructed yet (make_widget hasn't been called). So `updateWidgets` has no registered widget to update. By the time the widget is constructed (make_widget returns), the ChannelSets may not be pushed.

**Fix strategy:** In `ShuffleMatrixWidget::buildLayout()`, call `setChannelSets` directly if the ChannelSet members in the knob are their default (Chan_Black) values — detect this and pull from the Op. Alternatively, `make_widget` should trigger an initial sync. The cleanest fix: in `ShuffleMatrixKnob::make_widget()`, after constructing the widget, call `widget->syncFromKnob()` explicitly so the first open always reflects current ChannelSets.

A second approach: in `knob_changed` when `k == &Knob::showPanel`, explicitly call `updateWidgets()` after a zero-timer delay using `QTimer::singleShot(0, ...)`. But this is a workaround and the root-cause fix is more reliable.

### Anti-Patterns to Avoid
- **Calling `setValue()` from inside `buildLayout()`:** This fires `changed()` → `kUpdateWidgets` → `syncFromKnob()` → `buildLayout()` — infinite recursion. Identity init must write state via a non-firing path.
- **Clearing `_matrixState` in response to ChannelSet changes:** Explicitly prohibited by the locked decisions.
- **Using `btn->setText("X")` for X-mark in QSS:** Text content in QSS pseudo-states is not supported in Qt — use `paintEvent` or explicit `setText` calls synchronized with state.
- **Hiding in2 columns when in2 == none:** They must remain visible but disabled. Removing them from the layout causes layout reflow that confuses users.
- **Using `setHidden(false/true)` on NDK knobs to hide them:** Prefer `Knob::INVISIBLE` flag set during `knobs()` construction — dynamic hide/show of NDK knobs has undefined behavior in some Nuke versions.

## Common Pitfalls

### Pitfall 1: Recursive Layout Rebuild via setValue
**What goes wrong:** Calling `_knob->setValue(identityState)` inside `buildLayout()` fires `changed()`, which schedules `kUpdateWidgets`, which calls `syncFromKnob()`, which calls `clearLayout()` + `buildLayout()` again — infinite loop, stack overflow.
**Why it happens:** `setValue` is designed for user interactions, not internal initialization.
**How to avoid:** For identity initialization, write the state directly to the knob's internal `_matrixState` without calling `changed()`. Add a `ShuffleMatrixKnob::initializeState(const std::string&)` method that sets `_matrixState` without firing `changed()`. Then sync button states inline after returning from `buildLayout()`.
**Warning signs:** Nuke freezes or crashes on first node panel open.

### Pitfall 2: outGroup Capture Bug in Lambda
**What goes wrong:** The `buildGroup` lambda captures `groupLabel` by reference or the wrong value, causing all rows in both groups to identify as the same group during radio enforcement.
**Why it happens:** `groupLabel` is a local string; if captured by reference in a lambda that outlives the local scope, it becomes a dangling reference.
**How to avoid:** Always capture group-identifying strings by value in lambdas stored in buttons. Use `const std::string capturedOutGroup = groupLabel;` before the lambda.

### Pitfall 3: NDK Knob INVISIBLE Flag Behavior
**What goes wrong:** After hiding the four `Input_ChannelSet_knob` knobs with INVISIBLE, they may no longer participate in `knob_changed`, or their values may not serialize to the Nuke script, breaking load/save.
**Why it happens:** Some NDK flag combinations suppress `store()` calls or `knob_changed` callbacks.
**How to avoid:** Use `Knob::INVISIBLE` only, not combined with `DO_NOT_WRITE` unless you have an alternative serialization path. Verify the four channel set knobs still appear in the saved `.nk` script after adding INVISIBLE. If they disappear from the script, remove `DO_NOT_WRITE`.

### Pitfall 4: Programmatic ChannelSet Knob Update from QComboBox
**What goes wrong:** Calling `_knob->op()->knob("in1")->set_text(layerName)` from a Qt signal handler on the UI thread triggers a Nuke recook on the wrong thread, or the ChannelSet knob does not recognize the text format.
**Why it happens:** NDK's `ChannelSet` knob expects a specific string format (e.g. `"rgba"` for a layer, not a full channel name). Wrong format silently fails.
**How to avoid:** Test the exact string format that `Input_ChannelSet_knob` accepts (likely just the layer prefix like `"rgba"` or `"depth"`). Use `executeInMainThreadWithGuard` or `QMetaObject::invokeMethod` if the signal fires from a non-main thread. Alternatively, use NDK's `set_value(ChannelSet)` overload if available.

### Pitfall 5: const:0 / const:1 Column Serialization Parsing
**What goes wrong:** The current `_validate()` parser in `DeepCShuffle2.cpp` strips the `in1:` / `in2:` prefix before looking for `const:0`. If the serialized token is `outName:in1:const:0`, after stripping `in1:` the remainder is `const:0` — this works correctly. But if the UI writes `outName:const:0` (without group prefix), the parser must also handle it.
**Why it happens:** The const columns have no meaningful input group (they are not in1 or in2). The UI should write them with a group prefix (e.g. `in1:const:0`) by convention, matching the existing pattern.
**How to avoid:** In `buildLayout()`, when constructing the connect lambda for the const:0 / const:1 buttons, use `"in1"` as the input group string (these columns logically belong between in1 and in2). This ensures the serialized format remains consistent with what `_validate()` already parses.

### Pitfall 6: make_widget Timing vs. knob_changed Timing
**What goes wrong:** `knob_changed` with `showPanel` fires before `make_widget` is called. The widget does not exist yet when `updateWidgets` is called, so the initial ChannelSet push is lost. The widget opens showing default/stale channel headers.
**Why it happens:** Nuke's panel open sequence fires `knob_changed(showPanel)` → `updateWidgets()` (no widget yet) → `make_widget()` → widget constructed with stale knob ChannelSet state.
**How to avoid:** In `ShuffleMatrixKnob::make_widget()`, after constructing the `ShuffleMatrixWidget`, immediately push current ChannelSets into the widget before returning. This ensures the widget has correct state from its very first render. The `setChannelSets` + `syncFromKnob` pair suffices.

## Code Examples

### X-Mark Button via Custom paintEvent
```cpp
// ChannelButton.h — QPushButton subclass for colored channel toggle
class ChannelButton : public QPushButton
{
    Q_OBJECT
public:
    ChannelButton(QColor baseColor, QWidget* parent = nullptr)
        : QPushButton(parent), _baseColor(baseColor) {
        setCheckable(true);
        setFixedSize(22, 22);
        setFlat(true);
    }
protected:
    void paintEvent(QPaintEvent* event) override {
        QPainter painter(this);
        painter.setRenderHint(QPainter::Antialiasing);

        QColor fillColor = isEnabled() ? _baseColor : QColor(60, 60, 60);
        if (isChecked() && isEnabled())
            fillColor = fillColor.lighter(130);
        painter.fillRect(rect(), fillColor);

        painter.setPen(QPen(QColor(30, 30, 30), 1));
        painter.drawRect(rect().adjusted(0, 0, -1, -1));

        if (isChecked() && isEnabled()) {
            painter.setPen(QPen(Qt::white, 2));
            painter.drawLine(4, 4, 17, 17);
            painter.drawLine(17, 4, 4, 17);
        }
    }
private:
    QColor _baseColor;
};
```

## Open Questions

1. **NDK INVISIBLE flag behavior with KNOB_CHANGED_ALWAYS**
   - What we know: `Knob::INVISIBLE` hides the knob row from the panel UI. `KNOB_CHANGED_ALWAYS` ensures `knob_changed` fires on every change.
   - What's unclear: Whether combining both flags still allows `knob_changed` to fire when the knob's value is changed programmatically from the embedded QComboBox.
   - Recommendation: Test with a minimal case in Nuke. If INVISIBLE suppresses `knob_changed`, use a different flag combination or call `knob_changed` manually after the programmatic set.

2. **Best API for writing back to ChannelSet knob from QComboBox**
   - What we know: `Op::knob(name)` returns a `Knob*`. ChannelSet knobs have `set_value` but the exact overload for ChannelSet is not verified.
   - What's unclear: Whether `knob("in1")->set_text("rgba")` correctly sets the ChannelSet to `Mask_RGBA`, or if a different call is needed.
   - Recommendation: At task execution time, check the NDK source or an existing plugin that programmatically sets an `Input_ChannelSet_knob`. The simplest tested approach: write the layer name string via `knob->set_text()` and then call `Op::knob_changed(knob)` explicitly.

3. **Identity init: buildLayout vs. syncFromKnob timing**
   - What we know: Identity init must not cause recursive rebuild. It should fire only once (on first open with empty state).
   - What's unclear: Whether checking for empty state in `buildLayout()` is the right place, or whether `syncFromKnob()` before calling `buildLayout()` is safer.
   - Recommendation: Initialize in `syncFromKnob()` before calling `buildLayout()`. Check `_knob->matrixState().empty()` there, compute identity, write via `initializeState()`, then proceed to `buildLayout()` which will find non-empty state and skip init.
